# woke_novel 可视化前端完整功能清单

## 1. 文档目标

本文列出可视化前端从 MVP 到完整版本需要实现的全部功能。功能按模块拆分，并标注优先级，便于后续排期、开发和验收。

优先级定义：

- P0：第一版必须实现，没有它无法形成可用闭环。
- P1：强烈建议第二阶段实现，会显著提升可用性。
- P2：增强功能，适合稳定后逐步完善。
- P3：长期能力，适合桌面化、专业化或多用户化之后实现。

## 2. 总体功能范围

完整前端需要覆盖八类核心对象：

- 项目：创建、查看、继续、删除、元数据管理。
- 工作流：完整执行、断点续跑、单步执行、状态监控。
- 创意：多方案生成、比较、选择、补充。
- 文件：项目产物浏览、阅读、编辑、保存、diff。
- 章节：正文列表、字数统计、完整性检查。
- 日志：实时日志、历史日志、失败诊断。
- 导出：MD / TXT / EPUB 导出、导出前检查。
- 设置：语言、后端、环境检测、运行策略。

## 20. MVP 范围

第一版最小可用闭环建议包含：

1. 项目列表。
2. 新建项目向导。
3. 完整执行。
4. 断点续跑。
5. 单步执行。
6. 工作台实时日志。
7. 基础阶段时间线。
8. 文件树。
9. Markdown 阅读。
10. 章节列表。
11. 导出前检查。
12. MD / TXT / EPUB 导出。
13. 环境检测。
14. 设置语言和后端。
15. 统一错误提示。
16. 项目重命名同步。

MVP 验收标准：

- 用户可以在网页中创建项目。
- 用户可以启动干运行或真实运行。
- 用户可以看到运行日志和任务结果。
- 用户可以打开生成的项目文件。
- 用户可以继续中断项目。
- 用户可以导出小说成品。
- 失败时用户能看到错误和日志入口。
- 完整工作流将项目目录重命名后，前端能自动切换到新项目。

## 21. 第二阶段范围

第二阶段重点提升创作控制力：

1. 创意方案并排比较。
2. 方案选择后自动补充。
3. Markdown 编辑器。
4. 保存前 diff。
5. 文件自动备份。
6. 循环矩阵。
7. 产物健康度。
8. 章节完整性检查。
9. 诊断建议。
10. EPUB 拆章工具。

## 22. 第三阶段范围

第三阶段重点提升专业体验：

1. 人物结构化视图。
2. 状态文档事实板。
3. 章节质量检查。
4. 导出模板。
5. 项目备份恢复。
6. Tauri 桌面壳。
7. 文件选择器。
8. 系统通知。
9. 多主题。
10. 布局记忆。

## 23. 总结

完整可视化前端需要从“能运行工作流”逐步扩展到“能管理创作资产、诊断失败、编辑产物、检查章节、导出成书”。第一版应坚持闭环优先：项目、运行、日志、文件、导出必须跑通；第二版再强化创意比较、编辑和诊断；第三版再进入桌面化和专业创作辅助。
