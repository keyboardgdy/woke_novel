# woke_novel 可视化前端开发总览

## 1. 当前系统分析

woke_novel 目前是一个本地 Python CLI 工作流系统，核心边界非常清晰：

- `menu.py`：交互式菜单入口，负责语言、后端、项目、导出、工具和删除等用户操作。
- `run_workflow.py`：命令行工作流入口，提供 `init`、`loop`、`continue`、`single`、`session` 命令。
- `workflow_runner.py`：核心执行器，负责加载步骤模板、替换路径变量、调用 Claude CLI / Codex CLI、写入日志、更新断点。
- `path_resolver.py`：路径和模板变量解析，区分 `steps/` 与 `steps_en/`。
- `project_info.py`：维护 `projects/<项目名>/.project_info.json`，保存断点、轮次、幕次、创意方案等元数据。
- `project_structure.py`：创建项目标准目录。
- `projects/`：小说产物目录。
- `logs/`：运行日志目录，与 `projects/` 平级。

现有系统的价值不在于 UI，而在于已经具备稳定的“模板约束 + 多步骤编排 + 断点续跑 + 本地文件产物”核心能力。可视化前端应复用这些能力，不应重新实现小说生成逻辑。

## 2. 前端建设目标

可视化前端的第一目标不是做一个漂亮首页，而是把当前分散在 CLI、目录、日志和 JSON 断点中的信息组织成一个本地创作控制台。

必须优先解决：

- 项目可见：用户能看到有哪些项目、各自跑到哪里。
- 运行可控：用户能启动完整流程、断点续跑、单步执行，并看到任务状态。
- 文件可读：用户能打开产物、日志、章节和项目元数据。
- 失败可诊断：失败后能看到命令、返回码、stderr、最近日志和重跑入口。
- 导出可检查：导出前能确认章节数量、顺序、空章节和输出路径。

第二阶段再处理编辑器、diff、人物关系图、状态事实板、Tauri 桌面化等增强能力。

## 4. 当前代码事实

### 4.1 工作流命令

现有 CLI 支持：

```text
python run_workflow.py init
python run_workflow.py loop --project-name <project> --genre <genre> --option-count 3 --language zh --provider claude
python run_workflow.py continue --project-name <project> --language zh --provider claude
python run_workflow.py single <step> --project-name <project> --genre <genre> --language zh --provider claude
python run_workflow.py session <block> <s1,s2,...> --project-name <project>
```

注意：当前 `single` 命令没有 CLI 参数用于传入 `act_num`、`round_num`、`option_index`。如果前端 API 需要精确单步重跑循环步骤，后端应优先直接调用 `WorkflowRunner.run_step(...)`，或者扩展 `run_workflow.py single` 参数。

### 4.2 断点字段

`.project_info.json` 当前由 `ProjectInfo` 管理，核心字段包括：

- `project_name`
- `genre`
- `novel_size`
- `target_word_count`
- `created_at`
- `current_round`
- `selected_option`
- `ref_works`
- `novel_name`
- `last_step`
- `last_step_at`
- `last_step_act`
- `last_step_round`
- `last_step_option_index`
- `last_step_phase`
- `workflow_version`
- `option_count`
- `act_count`
- `chapter_counts`
- `total_chapters`

其中 `language` 和 `provider` 目前主要来自 `.menu_config.json`、命令参数或运行请求，不是所有项目天然持久化字段。API 可以在返回模型中提供这两个字段，但要标记为“派生值或默认值”。

### 4.3 日志规则

日志写入位置：

```text
logs/<project>/
```

当前步骤日志文件名大致为：

```text
<seq>_step_<step>[_A<act>]_R<round>_C<session>.log
```

日志内容包含 step、round、act、display_id、provider、session_uuid、time、attempts、prompt、每次 attempt 的 return code、elapsed、stdout、stderr。第一版 API 可以先读取日志文件和 tail 内容，不必一开始就解析成完整结构化事件。

### 4.4 外部 CLI 调用

Claude：

```text
claude @<temp_prompt> --session-id <uuid> --permission-mode bypassPermissions
claude @<temp_prompt> --resume <uuid> --permission-mode bypassPermissions
```

Codex：

```text
codex exec --json --skip-git-repo-check --dangerously-bypass-approvals-and-sandbox -
codex exec resume --json --skip-git-repo-check --dangerously-bypass-approvals-and-sandbox <session_id> -
```

前端必须明确显示当前后端和运行风险，尤其是真实运行会调用外部 CLI 且可能绕过交互确认。

### 4.5 导出与工具

`menu.py` 当前包含：

- 导出 MD。
- 导出 TXT。
- 导出 EPUB，需要 `ebooklib`。
- EPUB 拆章，输出到源 EPUB 所在目录下的 `Output/`。
- 删除项目，同时删除 `projects/<project>` 和 `logs/<project>`。

可视化 API 第一版可以复用这些实现，但建议抽成服务函数，避免从 Web 后端直接驱动交互式菜单。

## 5. MVP 开发顺序

建议按以下顺序落地：

1. 后端基础：FastAPI、健康检查、设置读取、环境检测。
2. 项目扫描：读取 `projects/` 和 `.project_info.json`。
3. 文件树：安全读取项目目录、Markdown、JSON、日志。
4. 任务执行：启动 `loop`、`continue`、`single`，维护 run_id、状态和互斥锁。
5. 实时日志：SSE 推送 `log_line`、`run_state`、`heartbeat`。
6. 工作台：项目状态、下一步建议、阶段时间线、日志面板。
7. 导出检查：章节自然排序、空章节、目标章节对比。
8. 导出执行：MD / TXT / EPUB。
9. 创意方案基础读取和选择。
10. 编辑器、diff、备份和诊断增强。

## 6. 第一版实现边界

第一版应坚持本地单用户模型：

- 服务只绑定 `127.0.0.1`。
- 前端不直接读写本地文件。
- API 不接受任意绝对路径写入。
- 同一项目同一时间只允许运行一个任务。
- 文件保存第一版可以只读，编辑放到第二阶段。
- 结构化步骤事件可以后置，先保证日志和任务状态可见。
- 不做远程多用户、云同步、在线协作。

## 7. 关键风险

- 项目重命名：创意选择后 `WorkflowRunner.rename_project()` 可能把初始项目目录重命名为小说名，API 需要感知项目名变化。
- 单步上下文：循环步骤单步执行需要 `act_num` 和 `round_num`，当前 CLI `single` 未暴露这些参数。
- 语言/后端字段：项目 JSON 不一定保存 `language` 和 `provider`，前端不能假设所有旧项目都有。
- 日志结构：当前日志是文本文件，不是事件数据库；第一版事件应由任务管理器产生，历史步骤事件可从日志和断点降级推断。
- 产物不确定：模型可能没有按模板写入预期文件，前端需要文件健康度检查。
- 权限参数：当前 Claude / Codex 调用参数偏自动化，UI 必须给出真实运行风险提示。

## 8. 验收闭环

MVP 达标标准：

- 可以在网页查看项目列表。
- 可以创建新项目并启动干运行或真实运行。
- 可以断点续跑已有项目。
- 可以单步运行普通步骤。
- 可以看到实时日志、历史日志和失败摘要。
- 可以浏览项目文件和章节。
- 可以执行导出前检查并导出 MD / TXT / EPUB。
- 失败时不需要用户进入文件夹手动找日志。
