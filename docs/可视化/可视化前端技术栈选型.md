# woke_novel 可视化前端技术栈选型

## 1. 选型结论

推荐采用“本地 Web 控制台 + Python API 服务 + 可选桌面壳”的技术路线。

核心技术栈：

- 前端：React + TypeScript + Vite。
- UI：Tailwind CSS + shadcn/ui + lucide-react。
- 状态与数据：TanStack Query + Zustand。
- 编辑器：Monaco Editor，轻量场景可替换为 CodeMirror。
- Markdown 渲染：react-markdown + remark-gfm。
- 后端 API：FastAPI + Pydantic。
- 实时状态：Server-Sent Events，必要时升级 WebSocket。
- 任务执行：Python subprocess + 后台任务管理。
- 测试：pytest + Vitest + Playwright。
## - 桌面化：第二阶段使用 Tauri。

这套组合的核心理由是：现有项目已经是 Python CLI 工作流，前端需要管理本地文件、运行本地命令、读取日志和展示长文本。FastAPI 能最小成本包装现有 Python 能力；React + Vite 能快速构建复杂工作台；Tauri 可以在后续提供桌面应用体验，而不需要第一版就引入完整桌面复杂度。

## 2. 架构建议

推荐目录结构：

```text
woke_novel/
├── app_server/
│   ├── main.py
│   ├── api/
│   ├── services/
│   ├── schemas/
│   └── runtime/
├── frontend/
│   ├── src/
│   ├── public/
│   ├── package.json
│   └── vite.config.ts
├── menu.py
├── run_workflow.py
├── workflow_runner.py
├── projects/
├── logs/
└── docs/
```

职责划分：

- `frontend/`：只负责界面、交互、状态展示，不直接读写本地文件。
- `app_server/`：包装现有 Python 模块，提供 HTTP API 和运行任务管理。
- `workflow_runner.py`：继续作为核心工作流执行器，避免重复实现流程逻辑。
- `projects/`：继续作为小说项目产物目录。
- `logs/`：继续作为运行日志目录，前端通过 API 读取。

第一版不要改变现有 CLI 的可用性。前端应作为新增入口，而不是替换 `menu.py`。

实现分层建议：

- 简单运行类接口可以先通过 subprocess 调用 `run_workflow.py`，这样风险最低。
- 项目扫描、文件树、章节检查、环境检测、日志读取应直接用 Python 服务函数实现，不需要绕 CLI。
- 需要传入 `act_num`、`round_num`、`option_index` 的精确单步重跑，当前 CLI 参数不完整，建议由 API 后端直接封装 `WorkflowRunner`。
- 导出和 EPUB 拆章逻辑当前在 `menu.py` 中，建议抽到独立 service，再同时供 CLI 菜单和 API 调用。

## 3. 前端选型

### 3.1 React + TypeScript + Vite

推荐原因：

- 适合构建“项目列表、工作流监控、文件浏览、编辑器、日志面板”这类状态丰富的工作台。
- TypeScript 能降低 API 字段、项目元数据、步骤状态映射的维护成本。
- Vite 启动快，适合本地工具开发。
- 生态成熟，后续接入 Monaco、Playwright、Tauri 都比较顺。

不推荐第一版使用 Next.js：

- 当前系统不是公网内容站，也不需要服务端渲染。
- Next.js 会引入路由、构建和部署复杂度，对本地工具收益不高。

### 3.2 Tailwind CSS + shadcn/ui

推荐原因：

- 能快速搭建专业、克制的工具型界面。
- 组件可复制进项目，不强依赖黑盒组件库。
- 适合做表格、弹窗、标签页、抽屉、命令面板、表单等工作台组件。

配套组件建议：

- Button、Input、Select、Tabs、Dialog、Sheet、Table、Toast、Tooltip。
- Progress、Badge、Separator、ScrollArea。
- Command 用于快速跳转项目、文件、步骤。

视觉原则：

- 信息密度高于普通网页。
- 状态色只用于运行状态、错误、警告和成功。
- 不做营销首页，启动后直接进入项目工作台。

### 3.3 lucide-react

推荐原因：

- 图标风格统一，覆盖运行、暂停、继续、导出、设置、日志、文件、警告等常见操作。
- 与 shadcn/ui 搭配自然。

建议图标用途：

- `Play`：启动或继续。
- `RefreshCw`：重跑步骤。
- `Pause`：暂停提示或停止任务入口。
- `FileText`：正文和 Markdown 文件。
- `FolderTree`：项目文件树。
- `Download`：导出。
- `AlertTriangle`：异常。
- `Settings`：设置。
- `Terminal`：日志。

### 3.4 TanStack Query

推荐原因：

- 项目列表、项目详情、文件内容、日志摘要都属于服务端数据。
- 支持缓存、刷新、错误状态和乐观更新。
- 比手写 `fetch + useEffect` 更稳。

适用数据：

- `GET /api/projects`
- `GET /api/projects/{name}`
- `GET /api/projects/{name}/files`
- `GET /api/projects/{name}/artifacts`
- `GET /api/projects/{name}/logs`
- `GET /api/settings`

### 3.5 Zustand

推荐原因：

- 适合保存轻量 UI 状态。
- 不需要 Redux 这类大型状态管理。

适用状态：

- 当前选中的项目。
- 当前打开的文件。
- 工作台布局状态。
- 日志过滤条件。
- 主题设置。
- 当前运行任务的本地视图状态。

### 3.6 Monaco Editor

推荐原因：

- Markdown、JSON、日志、配置文件都需要良好的编辑与查看体验。
- 支持差异对比，方便后续做“重跑前后 diff”。

适用场景：

- Markdown 产物编辑。
- `.project_info.json` 只读或受控编辑。
- 日志查看。
- 文件差异对比。

替代方案：

- 如果第一版希望更轻，可以使用 CodeMirror。
- 如果只做阅读器，先不用编辑器，使用 Markdown 渲染即可。

### 3.7 react-markdown + remark-gfm

推荐原因：

- 当前系统的大多数模板和产物是 Markdown。
- 需要支持表格、列表、标题、引用等常见格式。
- 可以与原文编辑器并排显示。

建议能力：

- 原文模式。
- 渲染模式。
- 分屏预览。
- 标题目录。

## 4. 后端选型

### 4.1 FastAPI + Pydantic

推荐原因：

- 与现有 Python 项目同语言，集成成本低。
- 类型化 schema 清晰，适合描述项目、步骤、文件、任务、日志。
- 自动生成 OpenAPI，方便前后端联调。
- 支持 SSE / WebSocket 和后台任务。

不推荐第一版使用 Flask：

- Flask 也能做，但 schema、类型提示、自动文档和异步扩展能力不如 FastAPI 顺手。

不推荐第一版使用 Node 后端：

- 现有核心逻辑都在 Python 中，Node 后端需要跨语言调用，复杂度上升。

### 4.2 API 设计

建议第一版接口：

```text
GET    /api/health
GET    /api/settings
PATCH  /api/settings

GET    /api/projects
POST   /api/projects
GET    /api/projects/{project}
DELETE /api/projects/{project}

GET    /api/projects/{project}/summary
GET    /api/projects/{project}/files
GET    /api/projects/{project}/files/content?path=...
PUT    /api/projects/{project}/files/content

POST   /api/projects/{project}/run/full
POST   /api/projects/{project}/run/continue
POST   /api/projects/{project}/run/single
GET    /api/projects/{project}/runs/{run_id}
GET    /api/projects/{project}/runs/{run_id}/events

GET    /api/projects/{project}/logs
GET    /api/projects/{project}/logs/{log_id}

POST   /api/projects/{project}/export
POST   /api/tools/split-epub
```

路径安全要求：

- 所有文件访问必须限制在 `projects/<project>/` 和 `logs/<project>/` 内。
- API 不接受任意绝对路径写入。
- 删除项目必须二次确认。
- 文件保存前返回 diff 或覆盖提示。

### 4.3 任务执行

第一版可以通过 subprocess 调用现有命令：

```text
python run_workflow.py loop --project-name <name> ...
python run_workflow.py continue --project-name <name> ...
python run_workflow.py single <step> --project-name <name> ...
```

推荐后端维护一个运行任务表：

- `run_id`
- `project`
- `command`
- `status`
- `started_at`
- `ended_at`
- `pid`
- `last_event_at`
- `return_code`
- `final_project_name`

任务状态：

- queued
- running
- succeeded
- failed
- cancelled

注意点：

- 同一个项目同一时间只允许一个工作流任务运行。
- 任务输出要写入日志，同时推送给前端。
- 子进程异常退出后，前端应能看到命令、返回码、最近日志。
- 完整工作流可能在创意选择后重命名项目目录，任务记录需要保存初始项目名和最终项目名。
- 如果第一版通过 subprocess 运行完整流程，结构化步骤状态主要来自任务管理器、日志文件和 `.project_info.json` 推断。

### 4.4 实时通信：优先 SSE

推荐第一版使用 Server-Sent Events。

原因：

- 工作流状态主要是后端推送给前端，前端不需要频繁向后端发送实时消息。
- SSE 比 WebSocket 简单，浏览器原生支持。
- 适合推送日志行、步骤状态、心跳和任务完成事件。

事件类型建议：

```text
run_started
log_line
step_started
step_completed
step_failed
artifact_detected
checkpoint_updated
project_renamed
run_succeeded
run_failed
heartbeat
```

什么时候升级 WebSocket：

- 需要在线终端交互。
- 需要前端实时发送 stdin。
- 需要复杂双向控制，例如运行中动态确认权限。

第一版不建议在浏览器里实现交互式终端确认。当前 `run_workflow.py` 的开篇确认已自动继续，Claude / Codex 调用也通过参数跳过交互确认；前端应把这件事显示为运行策略，而不是伪装成普通后台任务。

## 5. 文件与产物处理

### 5.1 文件树

后端负责扫描：

- `00_baseline`
- `01_plots`
- `02_guides`
- `02_output`
- `03_state`
- `04_characters`
- `CLAUDE.md`
- `.project_info.json`

返回字段：

- 文件名。
- 相对路径。
- 类型。
- 大小。
- 更新时间。
- 是否为空。
- 是否可编辑。

### 5.2 Markdown 与 JSON

Markdown：

- 默认可编辑。
- 保存前做路径校验。
- 保存后记录更新时间。

JSON：

- `.project_info.json` 默认只读。
- 人物档案 JSON 可以结构化展示。
- 如允许编辑，必须做 JSON parse 校验。

### 5.3 章节排序

正文目录必须使用自然排序：

- `正文v2.md` 应排在 `正文v10.md` 前。
- 导出前应再次排序和校验。

## 6. 测试选型

### 6.1 后端测试：pytest

重点测试：

- 项目扫描。
- `.project_info.json` 解析。
- 文件路径越界防护。
- 文件树生成。
- 导出前章节排序。
- run command 构造。
- 同项目任务互斥。
- 项目重命名后的任务记录和项目列表刷新。
- 现有步骤日志文件名解析。
- 带上下文单步运行的 service 封装。

### 6.2 前端测试：Vitest

重点测试：

- 步骤状态计算。
- 项目进度计算。
- 文件树转换。
- 日志过滤。
- API schema 适配。

### 6.3 端到端测试：Playwright

重点测试：

- 打开项目列表。
- 创建干运行项目。
- 查看工作流状态。
- 打开产物文件。
- 导出操作。
- 错误日志展示。

第一版至少应覆盖项目列表、新建项目、单步干运行、文件浏览四条路径。

## 7. 打包与运行

### 7.1 开发模式

推荐命令：

```text
python -m app_server.main
cd frontend
npm run dev
```

前端通过 Vite proxy 转发 `/api` 到本地 FastAPI。

### 7.2 生产模式

可选两种方式。

方式一：FastAPI 托管前端静态文件。

- 构建 `frontend/dist`。
- FastAPI 提供静态资源。
- 用户运行一个 Python 命令即可打开本地控制台。

方式二：Tauri 桌面壳。

- Tauri 启动前端窗口。
- 后台启动本地 FastAPI 服务。
- 更接近桌面应用体验。

建议先做方式一，再做 Tauri。

## 8. 不推荐方案

### 8.1 Streamlit

不推荐作为正式前端。

原因：

- 适合快速数据应用，不适合复杂文件管理、编辑器、长任务监控和工作台布局。
- UI 自定义能力有限。

可作为内部原型，但不适合作为最终产品。

### 8.2 Electron

第一版不推荐。

原因：

- 包体积和运行成本较高。
- 当前系统主要是本地 Python 工作流，Tauri 更轻。

如果后续需要更强 Node 生态或复杂桌面能力，再考虑 Electron。

### 8.3 纯静态前端

不推荐。

原因：

- 浏览器不能安全直接运行本地 CLI。
- 不能可靠读取和写入本地项目文件。
- 不能管理长时间运行的子进程。

### 8.4 直接改造 menu.py 成图形界面

不推荐。

原因：

- `menu.py` 适合作为交互式 CLI，不适合承载复杂可视化工作台。
- 前端需要实时日志、文件树、编辑器和多页面状态，应该由 Web UI 承载。

## 11. 技术路线总结

推荐路线是：

1. 保留现有 CLI 和 `WorkflowRunner`。
2. 新增 FastAPI 本地服务，包装项目、文件、日志、运行和导出能力。
3. 新增 React 工作台，提供项目管理、工作流监控、文件浏览和导出。
4. 使用 SSE 推送任务状态和日志。
5. 第一版跑通本地 Web 控制台后，再考虑 Tauri 桌面化。

这条路线能最大限度复用现有代码，同时为后续的编辑器、实时诊断、成书检查和桌面应用留下扩展空间。
