# woke_novel 可视化前端接口文档与数据结构

## 1. 文档目标

本文定义可视化前端与本地 API 服务之间的接口协议和核心数据结构。目标是让前端、后端和后续桌面壳可以围绕同一套契约开发。

推荐服务形态：

- API 服务：FastAPI。
- 接口前缀：`/api/v1`。
- 数据格式：JSON。
- 实时推送：Server-Sent Events。
- 文件路径：统一使用相对项目目录的 POSIX 风格路径，例如 `02_output/正文v1.md`。
- 运行方式：API 服务包装现有 `run_workflow.py`、`WorkflowRunner`、`projects/`、`logs/`。

重要约束：

- 前端不直接访问本地文件系统。
- API 不接受任意绝对路径写入。
- 所有项目文件访问必须限制在 `projects/<project>/` 内。
- 所有日志访问必须限制在 `logs/<project>/` 内。
- 同一项目同一时间只允许一个运行任务。
- 当前 `run_workflow.py single` 尚未暴露 `act_num`、`round_num`、`option_index` 参数。需要带上下文的单步运行应由 API 后端直接调用 `WorkflowRunner.run_step(...)`，或者先扩展 CLI 参数后再通过 subprocess 调用。
- 创意选择后，`WorkflowRunner.rename_project()` 可能把初始项目目录重命名为模型提取出的小说名。启动完整工作流的 API 必须能返回最终项目名变化事件。

## 2. 通用约定

### 2.1 基础 URL

```text
http://127.0.0.1:<port>/api/v1
```

开发默认端口建议：

- API：`8787`
- 前端：`5173`

### 2.2 通用响应

普通查询接口直接返回业务对象。

变更类接口返回：

```json
{
  "ok": true,
  "message": "created",
  "data": {}
}
```

### 2.3 错误响应

所有错误统一返回：

```json
{
  "ok": false,
  "error": {
    "code": "PROJECT_NOT_FOUND",
    "message": "Project not found: my_novel",
    "details": {
      "project": "my_novel"
    }
  }
}
```

常见错误码：

| 错误码 | HTTP 状态 | 说明 |
| --- | ---: | --- |
| `BAD_REQUEST` | 400 | 请求参数不合法 |
| `PROJECT_NOT_FOUND` | 404 | 项目不存在 |
| `FILE_NOT_FOUND` | 404 | 文件不存在 |
| `PATH_OUT_OF_SCOPE` | 400 | 路径越界 |
| `PROJECT_EXISTS` | 409 | 项目已存在 |
| `RUN_ALREADY_ACTIVE` | 409 | 项目已有任务运行中 |
| `RUN_NOT_FOUND` | 404 | 运行任务不存在 |
| `CLI_NOT_AVAILABLE` | 424 | Claude / Codex CLI 不可用 |
| `WORKFLOW_FAILED` | 500 | 工作流执行失败 |
| `EXPORT_FAILED` | 500 | 导出失败 |
| `UNSUPPORTED_FORMAT` | 400 | 不支持的导出格式 |

### 2.4 分页参数

列表接口如需要分页，使用：

```text
?limit=50&offset=0
```

响应：

```json
{
  "items": [],
  "total": 0,
  "limit": 50,
  "offset": 0
}
```

### 2.5 时间格式

统一使用 ISO 8601 字符串：

```text
2026-06-09T09:30:00+08:00
```

如果本地数据无时区，后端应按本机时区补齐或返回原始 ISO 字符串，同时保证前端可解析。

## 4. 核心数据结构

### 4.1 ProjectInfo

对应 `projects/<project>/.project_info.json` 的标准化展示模型。

```json
{
  "project_name": "my_novel",
  "novel_name": "长夜归墟",
  "genre": "玄幻",
  "language": "zh",
  "provider": "claude",
  "language_source": "settings",
  "provider_source": "settings",
  "novel_size": "中篇",
  "target_word_count": 300000,
  "current_round": 7,
  "selected_option": 2,
  "ref_works": "《A》《B》",
  "act_count": 3,
  "chapter_counts": [12, 10, 8],
  "total_chapters": 30,
  "last_step": "14",
  "last_step_name": "正文创作",
  "last_step_at": "2026-06-09T09:30:00+08:00",
  "last_step_phase": null,
  "last_step_act": 1,
  "last_step_round": 7,
  "last_step_option_index": null,
  "workflow_version": 2,
  "option_count": 3,
  "created_at": "2026-06-09T08:00:00+08:00"
}
```

字段说明：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `project_name` | string | 项目目录名 |
| `novel_name` | string/null | 最终小说名 |
| `genre` | string | 题材 |
| `language` | Language | 创作语言。当前旧项目 JSON 不一定保存，API 可从请求、运行记录或 `.menu_config.json` 派生 |
| `provider` | Provider/null | 默认后端。当前旧项目 JSON 不一定保存，API 可从请求、运行记录或 `.menu_config.json` 派生 |
| `language_source` | string | `project` / `run` / `settings` / `default`，说明语言字段来源 |
| `provider_source` | string | `project` / `run` / `settings` / `default`，说明后端字段来源 |
| `novel_size` | string | 小说规模 |
| `target_word_count` | number | 目标字数 |
| `current_round` | number | 当前正文循环轮次 |
| `selected_option` | number/null | 已选创意方案编号 |
| `ref_works` | string/null | 参考作品 |
| `act_count` | number/null | 幕次数 |
| `chapter_counts` | number[]/null | 每幕章节数 |
| `total_chapters` | number/null | 总章节数 |
| `last_step` | string/null | 最后完成步骤 |
| `last_step_name` | string/null | 步骤名称 |
| `last_step_phase` | string/null | 特殊 phase，如 `post_05b` / `post_17` |
| `last_step_act` | number/null | 最后步骤所属幕次 |
| `last_step_round` | number/null | 最后步骤所属轮次 |
| `workflow_version` | number | 工作流版本 |
| `option_count` | number/null | 创意方案数量，`loop` 开始时写入，旧项目可能缺失 |

### 6.2 创建项目

```text
POST /api/v1/projects
```

请求：

```json
{
  "project_name": "my_novel",
  "genre": "玄幻",
  "language": "zh",
  "provider": "claude",
  "novel_size": "中篇",
  "target_word_count": 300000,
  "user_description": "主角开局弱势，成长线清晰",
  "option_count": 3,
  "dry_run": false,
  "start_immediately": false
}
```

响应：

```json
{
  "ok": true,
  "message": "project created",
  "data": {
    "project_name": "my_novel",
    "project_path": "projects/my_novel"
  }
}
```

说明：

- `start_immediately=false` 时只创建目录和 `.project_info.json`。
- `start_immediately=true` 时创建后立即启动完整工作流，并在响应中返回 `run_id`。



- `act_num`、`round_num` 用于循环步骤上下文。
- `option_index` 用于步骤 01 / 02 的创意方案上下文。
- 如果通过 subprocess 调用现有 `run_workflow.py single`，这些上下文字段当前不会生效。MVP 可先限制单步运行为无上下文普通步骤；完整支持循环步骤重跑时，后端应直接调用 `WorkflowRunner.run_step(step, display_id, act_num=..., round_num=..., option_index=...)`，或扩展 CLI。

## 16. 后端实现注意事项

### 16.1 路径校验

后端必须执行：

- 将 `project` 校验为合法目录名。
- 将 `relative_path` 解析为绝对路径。
- 确认绝对路径仍在 `projects/<project>/` 内。
- 禁止 `..` 越界。
- 禁止保存到 `.git`、程序根目录和任意系统目录。

### 16.2 任务互斥

同一项目启动任务前检查：

- 是否已有 `queued` 或 `running` 任务。
- 如果有，返回 `RUN_ALREADY_ACTIVE`。
- 不同项目可并行，但第一版可以全局只允许一个任务，降低风险。

### 16.3 日志与事件

建议：

- 子进程 stdout/stderr 同时写入日志文件。
- SSE 从任务队列中读取结构化事件。
- 如果无法结构化识别步骤，第一版至少推送 `log_line` 和 `run_state`。

### 16.4 与现有 CLI 的映射

接口到命令的映射：

| API | CLI |
| --- | --- |
| `runs/full` | `python run_workflow.py loop --project-name <project> ...` |
| `runs/continue` | `python run_workflow.py continue --project-name <project> ...` |
| `runs/single` | `python run_workflow.py single <step> --project-name <project> ...` |

参数映射：

| API 字段 | CLI 参数 |
| --- | --- |
| `project_name` | `--project-name` |
| `genre` | `--genre` |
| `language` | `--language` |
| `provider` | `--provider` |
| `dry_run` | `--dry` |
| `option_count` | `--option-count` |
| `max_retries` | `--max-retries` |
| `user_description` | `--user-description` |
| `novel_size` | `--novel-size` |

当前 CLI 参数缺口：

| API 字段 | 当前 CLI 支持 | 建议 |
| --- | --- | --- |
| `act_num` | 否 | API 直接调用 `WorkflowRunner.run_step` 或扩展 `single` |
| `round_num` | 否 | API 直接设置 `runner.round` 后调用 `run_step` |
| `option_index` | 否 | API 直接调用 `run_step(..., option_index=...)` |
| `target_word_count` | 间接 | CLI 通过 `novel_size` 推导，API 可保留字段用于展示 |

### 16.5 数据兼容

当前 `.project_info.json` 可能缺少新字段。后端读取时应：

- 缺失字段返回 `null` 或默认值。
- 不因为旧项目缺少字段而报错。
- 写回时尽量保留未知字段。

## 18. 总结

这套接口把可视化前端需要的能力分成八类：系统设置、项目、步骤、运行任务、创意方案、文件、日志、导出工具。后端应优先保证本地文件安全、任务状态可靠、日志可追踪，再逐步增强创作资产解析和诊断能力。

第一版只要实现项目列表、新建/续跑/单步、事件流、文件浏览、日志和导出检查，就可以支撑一个完整可用的可视化工作台。
