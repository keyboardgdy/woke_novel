# woke_novel 可视化前端数据映射与文件规则

## 1. 文档目标

本文定义现有 woke_novel 文件、目录、步骤、日志与可视化前端之间的映射关系。它用于指导后端 API 的扫描、解析、校验和前端页面的数据展示。

核心原则：

- 前端不直接读写文件，由后端 API 统一处理。
- 后端只暴露相对项目路径，避免泄露或误写系统路径。
- 旧项目字段缺失时应降级展示，不阻塞页面加载。
- 原始产物优先保留，编辑和重跑必须有明确提示。

## 2. 根目录映射

| 本地路径 | 前端含义 | API 暴露 | 写入策略 |
| --- | --- | --- | --- |
| `projects/` | 小说项目根目录 | 是 | 仅通过项目接口创建/删除 |
| `logs/` | 运行日志目录 | 是 | 后端任务写入，前端只读 |
| `steps/` | 中文提示词模板 | 是 | 默认只读 |
| `steps_en/` | 英文提示词模板 | 是 | 默认只读 |
| `assets/style_presets_classified.json` | 中文题材预设 | 是 | 默认只读 |
| `assets/style_presets_classified_en.json` | 英文题材预设 | 是 | 默认只读 |
| `.menu_config.json` | 菜单配置 | 是 | 仅设置接口可写 |
| `run_workflow.py` | 工作流 CLI 入口 | 否 | 不由前端修改 |
| `workflow_runner.py` | 工作流核心 | 否 | 不由前端修改 |

说明：

- `language` 和 `provider` 当前主要由 `.menu_config.json`、CLI 参数或运行请求决定，并不是所有项目的 `.project_info.json` 都会持久化保存。
- API 展示项目时可以返回 `language` / `provider`，但应同时标明来源，避免旧项目被误判为缺字段。

## 3. 项目目录映射

单个项目目录：

```text
projects/<project>/
├── .project_info.json
├── CLAUDE.md
├── 00_baseline/
├── 01_plots/
├── 02_guides/
├── 02_output/
├── 03_state/
└── 04_characters/
```

| 子目录/文件 | 前端模块 | 内容类型 | 可编辑 | 说明 |
| --- | --- | --- | --- | --- |
| `.project_info.json` | 项目概览、工作台 | JSON | 受限 | 断点字段默认只读 |
| `CLAUDE.md` | 创作资产 | Markdown | P1 可编辑 | 项目级上下文 |
| `00_baseline/` | 创意与设定 | Markdown | P1 可编辑 | 创意、世界观、主轴、幕次框架 |
| `01_plots/` | 剧情 | Markdown | P1 可编辑 | 剧情提取、梗概、方向 |
| `02_guides/` | 写作指南 | Markdown | P1 可编辑 | 开篇和正文写作指南 |
| `02_output/` | 章节 | Markdown | P1 可编辑 | 正文输出，导出来源 |
| `03_state/` | 状态文档 | Markdown | P1 可编辑 | 连续性状态记录 |
| `04_characters/` | 人物 | JSON/Markdown | P1 可编辑 | 人物档案和关系 |

## 4. `.project_info.json` 字段映射

| 字段 | 前端位置 | 用途 | 缺失降级 |
| --- | --- | --- | --- |
| `project_name` | 项目列表、顶部项目切换 | 项目标识 | 使用目录名 |
| `novel_name` | 项目列表、导出标题 | 小说名 | 显示“未命名” |
| `genre` | 项目列表、工作台 | 题材 | 显示“未知题材” |
| `novel_size` | 项目概览 | 小说规模 | 显示“未设置” |
| `target_word_count` | 项目概览、章节统计 | 目标字数 | 不显示目标字数 |
| `current_round` | 工作台、循环矩阵 | 当前轮次 | 默认 `1` |
| `selected_option` | 创意方案页 | 已选方案 | 显示“未选择” |
| `ref_works` | 创意基准 | 参考作品 | 显示“无” |
| `act_count` | 工作台、循环矩阵 | 幕次数 | 根据文件推断或显示未知 |
| `chapter_counts` | 章节统计、导出检查 | 每幕章节数 | 仅展示已生成章节 |
| `total_chapters` | 章节统计、导出检查 | 目标总章节 | 使用 `sum(chapter_counts)` 或未知 |
| `last_step` | 项目列表、工作台 | 断点步骤 | 显示“未开始” |
| `last_step_at` | 项目列表 | 最后活动时间 | 使用目录更新时间 |
| `last_step_phase` | 续跑计算 | 区分步骤 18 阶段 | 由后端兼容推断 |
| `last_step_act` | 工作台 | 所属幕次 | 未知 |
| `last_step_round` | 工作台 | 所属轮次 | 使用 `current_round` |
| `workflow_version` | 兼容判断 | 工作流版本 | 按旧项目处理 |
| `option_count` | 创意方案、续跑计算 | 创意方案数量 | 默认 `3` |

写入规则：

- 可通过项目设置更新：`novel_name`、`genre`、`provider`、`language`、`novel_size`。
- 可通过创意选择更新：`selected_option`、`ref_works`、`novel_name`。
- 不建议前端直接更新：`last_step`、`last_step_at`、`last_step_phase`、`last_step_act`、`last_step_round`。
- 断点修复应作为高级功能单独实现，并要求二次确认。
- 如果允许保存 `provider` / `language` 到项目 JSON，应保留旧字段并只追加或更新安全字段，避免破坏旧项目兼容。

## 6. 文件命名与解析规则

### 6.1 章节文件

来源目录：

```text
projects/<project>/02_output/
```

排序规则：

- 使用自然排序。
- 从文件名中提取数字，例如 `v2` 小于 `v10`。
- 无数字文件排在数字文件后，按文件名排序。

章节标题提取：

1. 优先取 Markdown 第一个一级标题。
2. 没有一级标题时取第一个非空行。
3. 仍没有则使用 `第 N 章`。

字数统计：

- 中文：可按非空白字符粗略统计。
- 英文：可按单词统计。
- 第一版允许粗略统计，导出检查只用于发现明显异常。

### 6.2 创意方案文件

可能命名：

```text
00_baseline/创意方案_1.md
00_baseline/Creative_Proposal_1.md
```

解析字段：

- 方案编号：从文件名数字提取。
- 标题：优先取一级标题。
- 开篇预览：优先取 `## 开篇设想` 或 `## Opening Concept` 段落。
- 参考作品：从 `参考作品` 相关标题或正文中提取，失败时显示为空。

### 6.3 人物文件

来源目录：

```text
04_characters/
```

支持类型：

- JSON：结构化展示。
- Markdown：按普通 Markdown 展示。

JSON 展示规则：

- 保留原始 JSON。
- 尝试提取人物名、身份、目标、关系、变化线。
- 解析失败时不阻塞页面，显示原文和错误提示。

### 6.4 状态文档

来源目录：

```text
03_state/
```

第一版：

- 按 Markdown 阅读。
- 展示更新时间、字数、相关轮次。

第二阶段：

- 抽取角色状态、时间线、伏笔、地点、道具等事实板。

## 7. 日志映射规则

日志来源：

```text
logs/<project>/
```

现有步骤日志文件名：

```text
<seq>_step_<step>[_A<act>]_R<round>_C<session>.log
```

字段内容：

- step
- round
- act
- display_id
- provider
- session_uuid
- time
- attempts
- prompt
- 每次 attempt 的 return code、elapsed、stdout、stderr

前端展示：

- 历史日志列表。
- 运行中实时日志。
- 失败摘要。
- 最近错误行。

日志解析第一版要求：

- 能按文件更新时间排序。
- 能 tail 最后 N 行。
- 能区分当前任务日志和历史日志。
- 能展示 return code、命令和失败摘要，如果这些信息由任务管理器记录。
- 能识别现有步骤日志命名，提取 `step`、`act`、`round`、`session`。

日志解析第二阶段增强：

- 从日志中识别步骤开始、步骤完成、重试、超时。
- 生成结构化事件。
- 对常见错误给出诊断建议。

## 8. 页面数据源汇总

| 页面 | 主要数据源 | 备用/降级 |
| --- | --- | --- |
| 项目总览 | `projects/*/.project_info.json` | 目录名和目录更新时间 |
| 新建项目 | `assets/style_presets*.json`、设置 | 自定义题材输入 |
| 工作台 | `.project_info.json`、运行任务、日志 | 根据文件存在性推断 |
| 创意比较 | `00_baseline/创意方案_*.md` | 原文列表 |
| 创作资产 | 项目子目录文件树 | 只显示文件名 |
| 章节 | `02_output/*.md` | 空列表 |
| 导出 | `02_output/*.md`、`.project_info.json` | 只导出已存在章节 |
| 日志 | `logs/<project>/` | 显示无日志 |
| 设置 | `.menu_config.json`、环境检测 | 默认值 |

## 8.1 项目重命名规则

完整工作流在创意方案选择后会尝试从方案文件提取最终小说名，并调用 `WorkflowRunner.rename_project()` 重命名项目目录，同时迁移 `logs/<旧名>/`。

前端和 API 需要处理：

- 新建项目时的初始目录名可能不是最终目录名。
- `runs/full` 过程中可能出现 `project_renamed` 事件。
- 项目列表刷新时应以实际目录名为准。
- 如果重命名目标已存在，后端应展示失败提示，并继续保留原项目名。
- 所有长任务内部引用项目名时，应以运行器返回的当前项目名为准。

## 9. 文件健康度规则

健康度枚举：

- `ok`：文件存在且非空。
- `missing`：预期文件不存在。
- `empty`：文件存在但大小为 0 或仅空白。
- `stale`：文件时间早于关键上游文件。
- `placeholder`：疑似干运行占位内容。
- `unknown`：无法判断。

第一版检测：

- 是否存在。
- 是否为空。
- 更新时间。

第二阶段检测：

- 是否疑似占位。
- 是否早于上游文件。
- 是否章节过短。
- 是否 JSON 解析失败。

## 10. 编辑权限规则

| 文件类型 | 第一版 | 第二阶段 | 说明 |
| --- | --- | --- | --- |
| Markdown 产物 | 只读 | 可编辑 | 保存前备份 |
| JSON 人物档案 | 只读 | 可编辑 | 必须 JSON 校验 |
| `.project_info.json` | 只读 | 受限编辑 | 断点字段默认不可改 |
| `CLAUDE.md` | 只读 | 可编辑 | 提示影响后续生成 |
| `steps/*.md` | 只读 | 可选高级编辑 | 不建议普通用户改 |
| 日志文件 | 只读 | 只读 | 不允许编辑 |

## 11. 安全规则

后端必须执行以下校验：

- 项目名不能包含路径分隔符。
- 相对路径不能包含 `..` 越界。
- 解析后的绝对路径必须位于允许根目录内。
- 文件保存必须校验 hash，避免覆盖用户或任务刚写入的内容。
- 删除项目必须二次确认。
- 日志读取要支持截断，避免大文件阻塞。
- 导出文件名要做安全清洗。

## 12. 实施建议

第一版数据映射实现顺序：

1. 项目扫描。
2. `.project_info.json` 解析。
3. 文件树扫描。
4. Markdown 读取。
5. 章节自然排序。
6. 日志列表与 tail。
7. 导出检查。
8. 创意方案基础读取。

第二阶段再实现：

1. 创意方案结构化抽取。
2. 人物 JSON 结构化视图。
3. 文件健康度增强。
4. 文件编辑和备份。
5. 下游影响提示。
6. 日志结构化解析。
