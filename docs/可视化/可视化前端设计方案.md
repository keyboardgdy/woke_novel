# woke_novel 可视化前端设计方案

## 1. 前端目标

当前系统的核心能力已经比较完整：通过 `menu.py` 和 `run_workflow.py` 驱动 Claude CLI / Codex CLI，把 `steps/` 或 `steps_en/` 中的创作模板按顺序投喂给模型，并将产物写入 `projects/<项目名>/`。可视化前端的目标不是简单替代命令行菜单，而是把“长流程、长文本、多文件、多轮断点”的创作过程变成一个可观察、可编辑、可恢复的工作台。

前端应重点解决四类问题：

- 降低启动门槛：用户不用记命令、不用理解目录结构，也能创建项目、选择题材、选择模型后端并开始工作流。
- 增强过程感知：用户能看到当前跑到哪一步、产物是否生成、失败在哪里、日志是否异常。
- 提升创作控制：用户能预览创意方案、比较设定与人物、检查章节、必要时回退或重跑单步。
- 方便成品管理：用户能在一个界面内查看项目、导出 MD / TXT / EPUB、拆分 EPUB、管理日志和文件。

## 2. 必须包含的前端内容

### 2.1 项目总览

项目总览是进入系统后的第一屏，不建议做营销式首页。它应直接展示用户最关心的工作对象。

应包含：

- 项目列表：读取 `projects/` 下的项目目录。
- 项目状态：读取 `.project_info.json`，展示题材、小说名、规模、当前轮次、最后步骤、最后执行时间。
- 进度摘要：根据 `last_step`、`current_round`、`act_count`、`chapter_counts` 计算当前完成度。
- 快捷操作：继续执行、单步执行、查看产物、导出、删除。
- 异常提醒：缺少 `.project_info.json`、输出目录为空、断点信息不完整、日志存在失败记录。

设计巧思：

- 用“项目行 + 进度条 + 状态标签”的紧凑表格，而不是大卡片堆叠。小说项目可能很多，表格更适合反复扫描。
- 最后步骤不要只显示 `14`，应显示为“14 正文创作 / 第 N 轮 / 第 M 幕”。
- 给“可继续”和“需处理”两种状态不同视觉权重，可继续是常规主按钮，异常项目显示醒目的修复入口。

### 2.2 新建项目向导

新建项目应把当前 CLI 的多段输入整理成一个清晰流程。

应包含：

- 项目名称：对应 `projects/<项目名>/`。
- 创作语言：中文使用 `steps/` 和 `assets/style_presets_classified.json`，英文使用 `steps_en/` 和 `assets/style_presets_classified_en.json`。
- 模型后端：Claude CLI / Codex CLI。
- 小说题材：支持预设题材、细分主题、自定义输入。
- 小说规模：短篇 / 中篇 / 长篇 / 超长篇，对应目标字数。
- 用户补充说明：题材倾向、禁忌、主角设定、爽点要求等。
- 创意方案数量：对应 `option-count`。
- 干运行开关：对应 `--dry`，用于验证流程和目录写入。

设计巧思：

- 采用“左侧步骤导航 + 右侧表单”的向导，不要把所有配置塞在一个长表单中。
- 在最终确认页显示即将执行的等效命令，例如 `python run_workflow.py loop -p ...`，方便高级用户理解前端与 CLI 的关系。
- 后端选择旁边显示可用性检测：`claude --version` / `codex --version` 是否可执行。
- 干运行开关应有明显标识，因为它会生成占位产物但不调用外部模型。

### 2.3 工作流监控台

工作流监控台是可视化前端最核心的页面。它负责把当前“黑盒式长时间执行”变成可观察流程。

应包含：

- 步骤时间线：01 到 18 的完整步骤，以及 05b、11-16、17、18 这类循环/幕次步骤。
- 当前执行状态：等待、执行中、成功、失败、跳过、可重跑。
- 会话信息：Claude / Codex 后端、session display_id、尝试次数、耗时。
- 实时日志：读取 `logs/<项目名>/` 下的运行日志，支持过滤 stdout、stderr、错误、超时。
- 产物检查：当前步骤预期写入哪些文件，是否已经生成，文件大小和更新时间。
- 控制按钮：暂停提示、继续、重跑当前步骤、从断点续跑、进入单步模式。

设计巧思：

- 时间线不应只做一条横线。建议分成六个阶段：创意、世界/人物、主轴、开篇、循环创作、幕末整理。这样与当前 session 切分一致。
- 对循环步骤使用“幕次 × 章节轮次”的矩阵视图。例如行是幕次，列是 11-16，用户能直观看到第几幕第几轮卡住。
- 每个步骤状态旁显示“产物健康度”：文件存在、非空、最近更新时间、是否包含明显占位文本。
- 失败时自动展开最近一次 stderr 和失败命令，不要要求用户去日志目录里找。

### 2.4 创意方案比较与选择

步骤 01 会生成多个创意方案，步骤 02 会补充选中的方案。前端应该把这个节点做成可比较的选择界面。

应包含：

- 多方案并排预览：标题、题材、核心卖点、主角、开篇钩子、参考作品。
- 筛选与标记：收藏、排除、需要补充。
- 选择确认：写入 `selected_option`、`novel_name`、`ref_works`。
- 补充说明：允许用户在选择前追加偏好，再触发补充步骤。

设计巧思：

- 使用对比表而不是纯文本滚动。每一行比较同一维度，比如“主角欲望”“核心冲突”“长线悬念”。
- 给每个方案一个“风险提示”区，例如题材同质化、开篇不够强、设定过重、爽点延迟。
- 选择方案后在后续页面固定显示“当前创作基准”，减少用户忘记项目方向。

### 2.5 产物文件浏览器

现有项目产物分布在多个目录中，前端需要提供结构化浏览。

应包含：

- `00_baseline`：创意方案、补充方案、世界观、故事主轴、幕次框架。
- `01_plots`：剧情提取、剧情梗概、方向指导、章节梗概。
- `02_guides`：写作指南。
- `02_output`：正文章节。
- `03_state`：状态文档。
- `04_characters`：人物档案和关系文件。
- `CLAUDE.md`：项目级上下文。
- `.project_info.json`：项目元数据，只读展示，必要时提供安全编辑。

设计巧思：

- 左侧是目录树，右侧是 Markdown 阅读器，中间或顶部提供文件元信息。
- 正文目录应按章节号自然排序，避免 `v10` 排在 `v2` 前面。
- 对 JSON 人物档案提供结构化视图：人物名、身份、目标、关系、变化，而不是只显示原始 JSON。
- 文件阅读器支持“原文 / 渲染 / 差异”三种模式，为后续重跑前后的对比留空间。

### 2.6 编辑与人工干预

可视化前端不应只读。小说工作流需要允许用户在关键产物生成后做人工修订。

应包含：

- Markdown 编辑器：支持保存 `steps` 产物和项目产物。
- 安全编辑提示：哪些文件会影响后续步骤，例如世界观、人物档案、状态文档、CLAUDE.md。
- 修改记录：保存前后 diff，便于回溯。
- 重跑建议：编辑某个文件后提示可能需要重跑哪些后续步骤。

设计巧思：

- 编辑器旁边显示“下游依赖”。例如修改 `世界观.md`，提示可能影响人物、主轴、剧情和正文。
- 对状态文档提供“关键事实检查”视图，突出时间线、角色状态、伏笔、未解决冲突。
- 保存后不要自动重跑，应让用户明确确认，避免覆盖珍贵产物。

### 2.7 导出与工具

现有菜单已经支持导出小说和 EPUB 拆章，前端应把这些能力集中成“出版/工具”页面。

应包含：

- 导出格式：MD、TXT、EPUB、全部格式。
- 来源检查：统计 `02_output` 中的章节数量、缺失章节、空章节。
- 导出预览：书名、章节目录、文件路径。
- EPUB 拆章工具：选择本地 EPUB，输出到 `Output` 目录。
- 导出历史：展示最近导出的文件和时间。

设计巧思：

- 导出前做“成书检查”：章节数量是否等于 `.project_info.json` 中的 `total_chapters`，是否有异常短章节，是否存在重复标题。
- EPUB 导出页显示目录树预览，避免用户导出后才发现章节顺序错误。
- 成品文件旁提供“打开所在目录”或“复制路径”，但不要把它作为主要流程。

### 2.8 设置与环境检测

前端需要把当前 `.menu_config.json` 的能力可视化，并补充运行环境检测。

应包含：

- 当前语言：中文 / English。
- 当前后端：Claude CLI / Codex CLI。
- CLI 可用性：版本、PATH 检测、登录状态提示。
- Python 版本和依赖检测：`requirements.txt` 是否已安装。
- 工作目录检测：项目目录是否可信、是否可写。
- 日志与缓存：日志目录位置、清理策略。

设计巧思：

- 用“检测项列表 + 通过/警告/失败”表现环境，不要把错误藏在设置页面深处。
- 对失败项给出具体修复命令，但不自动执行涉及权限或安装的操作。
- 后端选择可以做成分段控件，旁边显示“推荐用于当前项目”的提示，例如 Codex/Claude 是否可用。

## 3. 信息架构建议

建议前端采用以下主导航：

- 项目：项目列表、创建、删除、摘要。
- 工作台：当前项目的流程监控、续跑、单步执行。
- 创作资产：世界观、人物、剧情、指南、正文、状态文档。
- 导出：MD / TXT / EPUB、EPUB 拆章、导出检查。
- 日志：运行日志、错误记录、耗时统计。
- 设置：语言、后端、环境检测。

单个项目内部建议采用以下二级导航：

- 概览：进度、题材、规模、最后步骤、关键产物。
- 流程：步骤时间线和运行控制。
- 设定：创意、世界观、人物、主轴。
- 章节：剧情梗概、写作指南、正文、状态。
- 文件：原始目录树。
- 导出：成品生成。

## 4. 关键页面布局

### 4.1 项目概览页

布局建议：

- 顶部：项目名、小说名、题材、当前后端、继续执行按钮。
- 中部：流程阶段进度条，按创意、世界、主轴、开篇、正文、整理分段。
- 下方左侧：最近产物和待处理事项。
- 下方右侧：项目元数据和导出状态。

巧思：

- 把“下一步应该做什么”作为页面主行动，而不是让用户自己判断。
- 例如显示：“下一步：从第 14 步正文创作继续，第 7 轮，第 1 幕。”

### 4.2 工作流运行页

布局建议：

- 左侧：步骤树和循环矩阵。
- 中央：当前步骤详情、预期输入、预期输出。
- 右侧：实时日志、错误、耗时。
- 底部：运行控制栏。

巧思：

- 每个步骤可以展开显示对应模板文件名，例如 `steps/14 正文创作.md`。
- 当步骤完成后，自动展示新生成文件的摘要和入口。
- 长时间执行时显示“最后日志更新时间”，避免用户误以为界面卡死。

### 4.3 创作资产页

布局建议：

- 左侧：资产类型导航。
- 中央：阅读器或结构化视图。
- 右侧：关联文件、下游步骤、修改建议。

巧思：

- 人物关系可以做成关系图，但不要只为了炫技。默认仍应保留可读表格。
- 正文阅读器可以显示章节字数、段落数、更新时间和是否低于预期字数。
- 状态文档可以抽取成“事实板”：角色状态、地点、伏笔、道具、时间线。

## 5. 视觉设计方向

这个系统是写作生产工具，不适合做成大幅插画、营销落地页或过度装饰的页面。视觉风格应偏“安静、清晰、可长时间工作”。

建议：

- 色彩：浅色或深色均可，但要控制对比，保留明确的状态色。主色用于导航和主行动，成功/警告/失败分别用绿/黄/红。
- 密度：比普通网页更紧凑，接近 IDE、文档管理器、任务面板。
- 字体：中文正文区域优先保证阅读舒适，代码、路径、日志使用等宽字体。
- 图标：运行、继续、暂停、重跑、导出、设置、日志、文件、警告等操作使用一致图标。
- 圆角：卡片和面板保持克制，避免大圆角造成玩具感。
- 动效：只用于运行状态、进度变化、错误展开，不做无意义装饰动效。

设计巧思：

- 状态色只用于状态，不用于大面积背景。这样失败和警告更容易被看见。
- “写作内容”和“系统日志”用明显不同的排版风格，避免用户误把日志当正文。
- 面向长文本编辑时，正文区域的行宽要受控，不能铺满超宽屏。

## 6. 与现有后端的集成方式

前端可以分三层接入现有系统。

### 6.1 最小接入

前端只作为命令封装器：

- 调用 `python run_workflow.py loop`
- 调用 `python run_workflow.py continue`
- 调用 `python run_workflow.py single`
- 读取 `projects/` 和 `logs/`

优点是改动小，能很快上线。缺点是实时状态需要从日志和文件变化中推断。

### 6.2 推荐接入

增加一个本地 API 层，例如 FastAPI：

- `GET /api/projects`
- `GET /api/projects/{name}`
- `POST /api/projects`
- `POST /api/projects/{name}/run`
- `POST /api/projects/{name}/continue`
- `POST /api/projects/{name}/single`
- `GET /api/projects/{name}/files`
- `GET /api/projects/{name}/logs`
- `POST /api/projects/{name}/export`

运行中的日志通过 WebSocket 或 Server-Sent Events 推送。

优点是前端结构清晰，后续可以扩展多端。缺点是需要把一部分 CLI 编排逻辑包装成服务接口。

### 6.3 深度接入

把 `WorkflowRunner` 的运行事件显式结构化：

- step_started
- step_retrying
- step_completed
- step_failed
- artifact_created
- checkpoint_updated
- export_completed

优点是前端可以精确显示状态，不依赖解析日志。缺点是需要改造核心运行器。

## 7. 技术选型建议

如果目标是本地桌面体验：

- 前端：React + TypeScript + Vite。
- UI：shadcn/ui 或同类组件库，搭配 lucide 图标。
- 编辑器：Monaco Editor 或 CodeMirror。
- Markdown：react-markdown / markdown-it。
- 后端：FastAPI。
- 实时通信：SSE 或 WebSocket。
- 桌面壳：后续可考虑 Tauri。

如果目标是轻量 Web 控制台：

- 前端仍可使用 React + Vite。
- 后端只绑定本机地址 `127.0.0.1`。
- 不处理远程多用户权限，先保证本机工具链稳定。

不建议第一版直接做复杂多用户、云端同步、在线协作。当前系统的核心假设是本地 CLI、本地文件、本地项目目录，前端应先服务这个事实。

## 8. 分阶段路线

### 第一阶段：可视化壳

目标：让用户不用命令行即可完成主要流程。

- 项目列表。
- 新建项目向导。
- 继续执行。
- 单步执行。
- 步骤列表。
- 导出。
- 日志查看。

### 第二阶段：创作工作台

目标：让用户能观察和干预创作过程。

- 步骤时间线。
- 实时日志。
- 产物文件浏览器。
- Markdown 阅读与编辑。
- 创意方案比较。
- 章节列表和正文预览。

### 第三阶段：质量与诊断

目标：让系统从“能跑”变成“好用、可控、可修复”。

- 章节完整性检查。
- 状态文档结构化摘要。
- 人物关系视图。
- 失败诊断。
- 重跑影响提示。
- 导出前成书检查。

### 第四阶段：桌面化与体验强化

目标：让非技术用户也能长期使用。

- Tauri 桌面应用。
- 一键环境检测。
- 项目备份与恢复。
- 主题切换。
- 文件变更历史。
- 更完善的错误引导。

## 9. 风险点

- 长流程阻塞：模型 CLI 调用可能持续很久，前端必须支持实时状态和超时提示。
- 文件写入不确定：模型可能没有按模板要求写入预期文件，需要产物检查机制。
- 重跑覆盖风险：前端必须在重跑前提示可能覆盖哪些文件。
- 编码问题：当前部分文档和 i18n 在某些终端显示异常，前端读取文件时要统一 UTF-8，并对异常文件给出提示。
- 权限风险：Claude / Codex 的权限参数较激进，前端应明确显示当前运行模式和风险，不要隐藏。
- 路径风险：项目名、导出文件名、删除操作都必须做路径校验，避免越界访问。
- 项目重命名风险：完整流程会在创意方案选择后尝试把项目目录重命名为最终小说名。前端必须同步当前项目路由、运行任务和日志目录，避免用户还停留在旧项目名。
- 单步上下文风险：正文循环里的单步重跑依赖幕次和轮次。当前 CLI `single` 参数不足以完整表达这些上下文，API 需要直接封装 `WorkflowRunner` 或扩展 CLI。

## 10. 一句话产品定位

这个前端应该是“小说流水线的本地创作控制台”：它不抢模型和模板的工作，而是把项目、步骤、文件、日志、断点、导出这些原本分散在命令行和目录里的信息，整理成一个可观察、可编辑、可恢复的写作工作台。
